Chapter 1

Executive summary

Seller Helper is the operations console packaged inside Vitrina Store at the route /seller-helper. It gives privileged sellers a single page to read the storefront's pulse — traffic, funnel, behaviour, security — and to act on it with curated, schema-validated AI recommendations and reversible quick fixes.

The product combines a deterministic rule engine (five thresholds expressed directly in SQL/TypeScript) with a large-language-model pass (OpenRouter primary, Gemini fallback) that runs against a tight JSON snapshot of the latest signals. Every output is fingerprinted, idempotent, and scoped to fields that can be undone — there is no destructive automation.

  • Surface: 7 tabs — Overview, Behavior, Funnel, Vitrina, AI Recommendations, Alerts, Security.
  • Engine: rule engine on rolling 15-minute, 2-hour and 7-day windows; LLM analysis with retries and provider fallback.
  • Auditability: alerts/recos carry a stable fingerprint per UTC day or hour; insert-on-conflict-do-nothing keeps re-runs safe.
  • Reversibility: 4 Vitrina quick fixes and 2 security quick fixes — block/unblock + 4 catalogue edits, no event deletion.
  • Privacy posture: session_key only, masked in UI, French-language operator copy on Security.
Chapter 2

Product context and audience

Seller Helper targets store owners and operations admins of the Algerian e-commerce front end (Vitrina Store). The audience expects French operational copy on sensitive flows (Security) and DA-denominated prices, while keeping global analytics in English (en-US locale) to align with web analytics conventions.

Operationally the page has two visible jobs:

  1. Diagnose — surface what is changing in traffic, conversion, behaviour and security right now.
  2. Act — apply small, reversible mutations (Vitrina catalogue tweaks, session blocks) without leaving the page.

The hero contains a live data badge (Live data · auto-refresh when events are flowing) and a sessions-per-15-minute counter. Two CTAs live to the right: Refresh re-reads the precomputed overview; Run analysis triggers the rule engine + LLM pass server-side.

Chapter 5

Access control and routing policy

The route guards itself before rendering a single byte to the client:

  1. auth.api.getSession() resolves the current Better Auth session. Missing session → redirect to /signin.
  2. The user's row is loaded from user to read the role column.
  3. Admin is role === "admin" || isPrivilegedAdminEmail(email). Non-admins receive a stub page with links back to /my-account and / — no telemetry payload ever reaches their browser.
  4. Only when admin is true does the page prefetch the overview, alerts, dismissed alerts and Vitrina recommendations in parallel via Promise.all.

Errors during the prefetch are captured as a string and rendered as a non-blocking banner; the dashboard still hydrates with whatever data was available.

Chapter 6

Data foundation: events, alerts, security blocks, applied actions

All Seller Helper computations rely on five tables:

  • sales_micro_event — the storefront micro-event log (pa_* events). Columns used: session_key, event_name, payload_json, page_path, product_local_id, created_at.
  • conception_alert — persistent alerts emitted by the rule engine and the LLM. Each row has a unique fingerprint, severity, optional metadata JSON and an affected_sessions_estimate.
  • conception_recommendation — long-form recommendations with priority, impactLabel, analysis, recommendation, confidence and an evidenceJson trail.
  • conception_security_block — active session blocklist. The unique key is session_key with a nullable lifted_at field — an open block has lifted_at IS NULL.
  • seller_helper_applied_action — audit log of operator actions taken inside the Seller Helper (Vitrina quick fixes, security blocks, alert resolutions, AI recommendations marked as applied). Indexed on occurred_at, product_local_id and kind. Powers the Timeline checkpoint overlay.

The canonical event vocabulary lives in src/server/conception/event-contract.ts:

export const STORE_EVENT = {

Anything outside this whitelist is intentionally invisible to the dashboard.

Chapter 7

Dashboard (Overview) tab

The Overview tab is the landing experience. It composes four panels:

  • KPI strip (top, 4 cards) — labels, values and a delta line marked positive/negative against the previous period. Drives the "is the store healthy?" judgement in under a second.
  • Traffic & Sales (24 h) — a custom SVG TrafficChart driven by normalised hourly counts from trafficHourlyNormalized(). Counts are scaled to the local max so the visual is stable regardless of absolute volume.
  • Devices — distribution slices parsed from the device field embedded in payload_json (mobile / tablet / desktop residual).
  • Top performing pages — top page paths by view count, joined with their add-to-cart events to compute a per-page conversion rate.

An empty-data fallback (No metrics available yet, No pages with recorded traffic) is rendered whenever the query returns zero rows — the dashboard never breaks on a fresh deployment.

Chapter 8

Timeline tab

The Timeline tab plots any combination of metrics over time. The X axis is always time; the Y axis is whatever the operator picks. Two Y axes are used simultaneously so volume (counts) and quality (percentages) can be compared on the same chart. Overlaid on top of the curves are checkpoint markers — small flag-shaped pins that mark the exact time an action was taken in any of the recommendation surfaces.

Controls

  • Range — 24h (hourly buckets), 7d / 30d / 90d (daily buckets). Buckets are aligned to the current UTC hour or UTC day.
  • Scope — Whole store, or a specific product. Per-product mode uses the same alias resolution as the heatmap (storefront product ID + DB hash aliases).
  • Metrics — multi-select chips. The default set is views, add-to-carts, and purchases. Hovering each chip shows the metric's exact definition.

Server pipeline

  1. bucketsForRange aligns the current time to UTC hour/day and emits N evenly-spaced bucket boundaries.
  2. loadEventCounts runs one GROUP BY query on sales_micro_event with date_trunc and per-event filters (FILTER WHERE) producing one row per bucket and per metric counter.
  3. For per-product scope, an IN filter on product_local_id uses the alias set returned by getCatalogProductAliasIds().
  4. Rate metrics (conversion rate) are computed at the bucket level — and at the window level for the KPI strip (sum of numerator / sum of denominator).
  5. listAppliedActionsInRange reads the seller_helper_applied_action audit table in the same window. Per-product timelines include both product-bound and store-wide checkpoints so operators see all signals competing for attribution.

Below the chart, a three-card KPI strip surfaces, per active metric, the total (or window rate for percent metrics), the average per bucket, the peak value and the peak bucket label. The chart itself is a custom SVG line chart with hover tooltip and dual Y-axes — count axis on the left, percent axis on the right — so volume and rate metrics can coexist without distorting each other. Checkpoints sit above the curves with a small color-coded legend that summarizes how many of each kind occurred inside the visible window.

Chapter 10

Conversion Funnel tab

The funnel computes nested-subset counts over the last 7 days. The CTE is intentionally written so that each level is the intersection with the previous one — i.e. n_cart only counts sessions that also had a pa_product_view.

WITH pe   AS (SELECT DISTINCT session_key FROM sales_micro_event WHERE event_name = 'pa_product_view'   AND created_at >= since),
     cart AS (SELECT DISTINCT session_key FROM sales_micro_event WHERE event_name = 'pa_add_to_cart'    AND created_at >= since),
     chk  AS (SELECT DISTINCT session_key FROM sales_micro_event WHERE (event_name = 'pa_begin_checkout' OR page_path LIKE '%checkout%') AND created_at >= since),
     fin  AS (SELECT DISTINCT session_key FROM sales_micro_event WHERE event_name = 'pa_purchase'       AND created_at >= since)
SELECT
  (SELECT COUNT(*) FROM pe)                                                                                        AS n_product,
  (SELECT COUNT(*) FROM pe JOIN cart USING (session_key))                                                          AS n_cart,
  (SELECT COUNT(*) FROM pe JOIN cart USING (session_key) JOIN chk USING (session_key))                             AS n_checkout_path,
  (SELECT COUNT(*) FROM pe JOIN cart USING (session_key) JOIN fin USING (session_key))                             AS n_final;

Friction items are then computed from the deltas between adjacent steps. The same numbers drive the recommendation engine — see the next chapter on the rule engine for the exact thresholds.

Chapter 11

Vitrina Recommendation tab

Vitrina is the merchandising lens. It enumerates the catalogue, builds a per-product interaction snapshot over the last 7 days, computes deterministic tips, ranks the products, and renders them in two views: a featured Swiper carousel and a filtered grid with search / category / sort controls.

Pipeline

  1. Load catalogue rows from productsTable joined with categoryTable; cross-reference with the storefront product IDs from the catalogue helper.
  2. Aggregate signals per storefront product over 7 days using a single GROUP BY query (views, hovers, clicks, add-to-carts, option selects, image views, specs interactions) plus a detail pass for view dwell time, scroll depth, click Y%, hover Y%, selected colours and review interactions.
  3. Build display snapshot — title hints (colour, price, brand), descriptionLength, rating, stock.
  4. Build interaction snapshot — window-normalised metrics, avg click Y%, avg hover Y%, top selected colours, viewToCartRate, clickToCartRate.
  5. Build tips — see the heuristics table below.
  6. Build quick fixes — at most four whitelisted IDs.
  7. Score and rank by opportunity score (priority + traffic weight + friction weight).

Chapter 13

Alerts tab

The Alerts tab consolidates active and dismissed incidents. Each row in conception_alert renders as a card carrying severity (critical / high / medium / low), title, description, optional detail, and an "Analyze in detail" action that calls /api/admin/conception/alerts/detail.

The detail endpoint runs analyzeConceptionAlertById() — a deterministic, per-type deep-dive function that produces:

  • Key indicators — current vs reference values for the rate / volume / error count.
  • Deviations — observed value vs baseline with a tone (critical/high/medium/low) computed from the delta.
  • Clues — qualitative pointers (e.g. "Loss often concentrates between cart and checkout initiation").
  • Fix steps — actionable checklist tailored to the alert family.

The dismissal flow accepts a disposition of "resolved" or "ignored", persisted on the alert row; the right column lists the most recent 12 dismissed alerts so operators can compare incident lifetimes.

Security tab

The Security tab is built from a dedicated brief computed by buildConceptionSecurityBrief():

  • 4 KPI cards — Bots détectés, Sessions bloquées, Tentatives de fraude, Score de sécurité.
  • Score formula card — the formula is shown verbatim to operators (see below).
  • Threat activity (24 h) — hourly series of events emitted by risky sessions; rendered through ThreatActivityChart.

The visible score formula:

Score = 100 − min(30, bots/sessions×35) − min(18, blocages/sessions×20) − min(24, fraudes×4) − min(16, pic horaire/80×16), borné entre 0 et 100.

Chapter 15

Rule engine — full catalogue

The deterministic rule engine lives in src/server/conception/analyze.ts and runs on every Run analysis click. Each rule emits an alert with a unique fingerprint so re-runs don't duplicate.

CONVERSION_DROP critical

Window: 7 days vs previous 7 days

Trigger: rateNow < 0.8 × rateOld (and rateOld > 0.001)

Fingerprint: Daily — one alert per UTC day.

Conversion rate has dropped more than 20% versus the rolling reference. Detail captures both rates so operators can compare instantly.

TRAFFIC_SPIKE high

Window: Last 15 minutes vs 90-minute baseline

Trigger: events15m > baseline15 × 4

Fingerprint: Hourly — one alert per UTC hour.

Event density at least 4× the 90-minute average over a 15-minute window. Hourly fingerprint avoids spam during sustained spikes.

CART_ABANDON_MASS medium

Window: Rolling 2 hours

Trigger: cart2h ≥ 8 AND cartAbandon2h ≥ 0.8

Fingerprint: Hourly.

At least 8 sessions added to cart in 2 hours and 80%+ never reached pa_purchase. affectedSessionsEstimate = cart2h − final2h.

JS_ERROR_BURST high

Window: Rolling 2 hours

Trigger: jsErrorSessions / sessionsCheckout2h ≥ 0.05

Fingerprint: Hourly.

More than 5% of checkout sessions raise pa_js_error events — usually a regression on the checkout page.

PERF_SLOW low

Window: Rolling 2 hours

Trigger: slowNavSessions + lcpSlowSessions ≥ 5

Fingerprint: Hourly.

Five or more sessions show slow navigation or LCP > 4 s, often image weight on the product page.

Chapter 16

LLM strategy and provider fallback

The LLM pass is orchestrated by runConceptionLlmAnalysis(). Its goal is to produce a French operator summary plus structured alerts and recommendations grounded in the current telemetry snapshot, never inventing metrics that aren't in the payload.

Provider chain

  1. OpenRouter (primary) — model picked from CONCEPTION_OPENROUTER_MODEL → ASSISTANT_FREEFLOW_MODEL → google/gemini-2.0-flash-001. JSON mode is enforced via response_format: { type: "json_object" }. Up to 3 attempts with 1.2 s × attempt back-off on 429/5xx.
  2. Gemini 2.0 Flash (fallback) — used when OpenRouter fails with a 402 (insufficient credits) or when CONCEPTION_LLM_GEMINI_FALLBACK=true. Calls generativelanguage.googleapis.com with responseMimeType: "application/json".

Validation pipeline

  1. extractJsonObject() tolerates wrapping prose / markdown fences.
  2. normalizeAnalysisPayload() accepts both English and French keys (alerts/alertes, recommendations/recommandations, analysis/analyse) and coerces severity / priority through fuzzy matching (crit, haute, moy).
  3. llmAnalysisSchema (Zod) enforces severity ∈ {critical, high, medium, low}, confidence ∈ [0, 100], max array sizes (6 alerts, 8 recommendations), and bounded string lengths.
  4. mapLlmOutput() appends day-fingerprints (LLM-ALERT-…-2026-05-12) and serialises metadata_json/evidence_json for traceability.

Determinism levers

  • Temperature is fixed at 0.2.
  • max_tokens is capped at 2 200 for OpenRouter.
  • The LLM only ever sees: latest overview, top 6 KPIs, funnel + summary + friction items, top 6 pages, devices, security brief and the analyze signals.

Chapter 18

Vitrina heuristics, scoring and ranking

Vitrina ranking is a three-component score in opportunityScore():

opportunityScore = priorityWeight + trafficWeight + frictionWeight

priorityWeight  = Σ (4 - IMPORTANCE_RANKS[tip.priority])     // high=3, medium=2, low=1
trafficWeight   = min(12, log2(views + 2) × 2)
frictionWeight  = (viewToCartRate != null && viewToCartRate < 0.1)
                  ? (0.1 - viewToCartRate) × 40
                  : 0

Ties are broken by the strongest tip priority (lowest IMPORTANCE_RANK). The list is then filtered:

  • Default view — any product with at least one tip.
  • actionableOnly — products with at least one high or medium tip.

The featured Swiper at the top of the tab shows the top three opportunity-ranked products; the grid below allows search by title/category/brand/slug, a category filter, and six sort modes (opportunity / interaction / merchandising priority — high or low).

Chapter 19

Quick fixes (Vitrina and Security)

Quick fixes are the only mutations the dashboard can apply. They share a strict contract:

  1. The client sends a structured payload (a list of { id, label, summary, context }).
  2. The server validates each id against a whitelist of VitrinaQuickFixId or ConceptionSecurityQuickFixId.
  3. For Vitrina, the server re-reads the recommendation, so the actual fixes applied must match what was offered — no smuggling of new fix IDs.

Chapter 20

Refresh model and side effects

The dashboard maintains three explicit state-flow primitives in useConceptionAdminData:

  • refresh() — re-fetches the overview + alert lists + Vitrina list without touching the LLM. Cheap and idempotent.
  • runAnalyze() — calls the analyze endpoint, which runs the rule engine, the LLM analysis (with provider fallback), and re-lists Vitrina. The hero button shows Analyzing… while busy. Returns the count of inserted alerts/recommendations and the LLM summary (if any).
  • dismissAlert / dismissRecommendation — optimistic UI dismissal with a server confirmation.

The hero status badge switches between Live data · auto-refresh (when overview.hasEventData is true) and Waiting for data · auto-refresh. The page is marked export const dynamic = "force-dynamic" so it never serves stale SSR cache.

Chapter 21

Privacy, integrity and operational policies

The following policies are enforced in code, not just in documentation. Each is observable in the source files listed below.

Privileged access only

/seller-helper resolves the current Better Auth session, joins user.role, and unions with the privileged admin email list. Anything that is not admin sees a stub page with links back to /my-account and / — no metrics are shipped to the client.

Server-rendered data, hydrated once

The page server-fetches the conception overview, active alerts, dismissed alerts and Vitrina recommendations in parallel (Promise.all). The client receives a typed ConceptionAdminInitialData payload and uses it as the initial state of useConceptionAdminData — no waterfall on first paint.

Schema-validated LLM output

Every LLM response goes through extractJsonObject, a permissive normaliser (English/French keys), then a Zod schema with hard enums (severity, priority) and clamped numeric ranges. If validation fails, the response is rejected — never written to the DB.

Idempotent, deduplicated writes

Alerts and recommendations use stable fingerprints (CONVERSION_DROP-2026-05-12, REC_FUNNEL_CART_CHECKOUT-2026-05-12). Inserts are onConflictDoNothing against the fingerprint, so re-running the analysis job is safe.

No raw PII in micro-events

session_key is an opaque identifier. The security brief explicitly notes: 'Les identités client sont dérivées des clés de session — aucune adresse IP n'est stockée dans les micro-événements.' Display identities are masked (sliced + ellipsis).

Bounded prompts and JSON-only contract

Both the conception system prompt and the Vitrina prompt force a single top-level JSON object with explicit keys, capped arrays (≤ 6 alerts, ≤ 8 recommendations, ≤ 3 Vitrina products), bounded string lengths and confidence ∈ [0, 100]. Temperature is fixed at 0.2.

Provider fallback with retry budget

OpenRouter is tried up to 3 times with exponential backoff on 429/5xx. A 402 (insufficient credits) automatically routes the same prompt to Gemini, unless CONCEPTION_LLM_GEMINI_FALLBACK overrides this default.

Localised seller voice

Dashboard copy is in English (en-US), but recommendation prose written by the LLM and the entire Security tab are forced to French — the Algerian merchant audience reads operational text in French and DA prices.

Quick fixes are scoped and reversible

Vitrina quick fixes can only mutate four whitelisted fields (jomla_price, description.colors, Availability, Quality). Security quick fixes can only block or unblock a session — never delete events.

Operationally, the recommended runbook is:

  1. Run a daily Run analysis per store, or trigger it from a scheduled job hitting the same endpoint.
  2. Treat fingerprints as authoritative deduplication keys; never alter them in migrations.
  3. If OpenRouter credit is depleted, the dashboard remains functional through Gemini fallback — monitor the llmModel field returned from analyze to know which provider answered.
  4. When extending the prompts, update the Zod schema and the normaliser at the same time; both must accept the new shape.
