Chapter 1

Executive summary

Vitrina Store is a production-grade e-commerce experience built on Next.js 15 and React 19, combining a rich merchandising front end, authenticated accounts, Algerian-market checkout (wilaya/commune), Chargily payments, and an AI-assisted shopping layer with multilingual responses and telemetry for continuous improvement.

This document narrates the product from conception through architecture, data design, operational concerns, and a structured catalog of use cases suitable for stakeholders, engineers, and auditors. Diagrams use plain figures so they render reliably in PDF engines.

  • Primary runtime: Node.js serverless (Vercel-style) with Route Handlers.
  • Persistence: PostgreSQL via Drizzle ORM.
  • Auth: Better Auth with Google OAuth and credential flows.
  • State: Redux Toolkit for cart, wishlist, quick view, product-details UI.
  • Intelligence: Google Gemini and/or OpenRouter-backed flows with catalog grounding.
Chapter 2

Vision, positioning, and conception

Vitrina targets a modern, mobile-first storefront with editorial polish: large imagery, confident typography, and fast interaction patterns (quick view, sticky assistant, expressive product pages). Conception emphasized three pillars: trust (clear pricing modes, policies, reviews), speed (client caching patterns, optimistic UI), and insight (shopping sequences + assistant telemetry).

The name and visual language evoke warmth and energy—orange accents on deep navy—while keeping commerce flows conventional so new shoppers remain oriented.

Design principle: progressive disclosure. Advanced modes (jomla pricing, assistant) enhance the core journey without blocking basic browse-to-buy.
Chapter 3

Stakeholders and personas

Guests discover the catalog, compare prices, and may purchase depending on checkout configuration. Registered users retain wishlists, reviews, and personalized surfaces. Operators monitor sequences and assistant metrics. Developers extend Drizzle models, API routes, and UI modules within the App Router structure.

Chapter 4

Functional scope overview

Major domains include catalog browsing, product detail, cart, checkout + payment, wishlist, authentication, blogs and static content, feedback and reviews, admin sequence analytics, and AI assistants (global + product-scoped).

Chapter 6

Solution architecture

The system follows a modular monolith pattern: a single deployable Next.js application containing UI, API routes, and server libraries. External systems are limited to well-bounded integrations (payments, LLM providers, OAuth).

Edge vs Node: Middleware performs a narrow cookie presence check for matched paths; sensitive authorization should still occur in server components or route handlers per Better Auth guidance.

Chapter 7

Frontend architecture (App Router)

Routes live under src/app/(site) with nested layouts supplying session, theme, Redux Provider, cart persistence, modals, and assistant shells. Pages remain mostly server components where feasible; interactive islands mount client components for cart, swiper carousels, and assistants.

  • Layouts: shared chrome, fonts, analytics hooks (as present).
  • Composition: components/ tree mirrors domains: Shop, Checkout, Cart, Common.
  • Styling: Tailwind CSS utility system with design tokens in tailwind config.
Chapter 8

State management

Redux Toolkit configures reducers for quick view, cart, wishlist, and product-details slices. Selectors are typed via useAppSelector. Cart persistence bridges storage and the store to avoid hydration mismatch and flicker.

Chapter 9

Authentication and sessions

Better Auth is instantiated in src/server/lib/auth.ts with Drizzle adapter, custom session enrichment (e.g., phone, image lookups), Google provider, and trusted origins derived from environment. Client helpers live in src/lib/auth-client.ts.

Chapter 10

Commerce, pricing modes, and inventory

Products carry list and optional promotional (jomla) prices with business rules documented in schema comments. Availability uses numeric stock representation suitable for extension to quantity-based fulfillment.

Chapter 11

Checkout, logistics fields, and payments

Checkout captures Algerian administrative geography (wilaya, commune) and optional customer notes. Payment initiation occurs server-side through Chargily’s REST API; the browser is redirected to hosted checkout, minimizing PCI scope on Vitrina infrastructure.

Chapter 12

AI assistants: behavior and safeguards

The assistant route loads the in-repo catalog (shopData), normalizes queries (including transliteration), optionally calls Gemini 2.0 Flash and/or OpenRouter models, merges LLM scores with deterministic signals, caches hot responses for five minutes, and returns structured matches plus a natural-language summary. Missing API keys produce clear, localized guidance rather than opaque failures.

Telemetry captures request identifiers, provider metadata, cache status, matched identifiers, and click-through events for offline evaluation.

Chapter 13

Analytics: shopping sequences

The shopping_sequence entity records session-keyed funnels with trigger labels, lifecycle status, and timestamps for product visits and endings. Admin APIs expose these rows for operational review.

Chapter 14

Data architecture and persistence

PostgreSQL holds canonical commerce entities, Better Auth tables, reviews, feedback, sequences, and assistant telemetry. Drizzle migrations under drizzle/ version additive changes.

Chapter 15

API surface (summary)

Server routes are implemented as Next.js Route Handlers returning JSON or redirects. The appendix lists every discovered route file in this repository snapshot.

Chapter 16

Security, privacy, and compliance posture

Secrets never ship to the client bundle. OAuth uses Google web client credentials. Payment secrets are server-only. Telemetry may contain search text—treat as sensitive operational data with retention controls.

Chapter 17

Deployment model

Typical deployment targets Vercel or compatible platforms with environment variables mirroring .env.example. Database connectivity uses pooled Postgres URLs suitable for serverless drivers.

Chapter 18

Testing strategy (recommendations)

Recommended layers: unit tests for pure utilities (price math, normalization), contract tests for Route Handlers with mocked Drizzle, Playwright smoke for checkout happy-path in staging, and periodic load tests on assistant endpoints.

Regression matrix (starter). Cart persistence across reload; assistant with keys absent/present; OAuth callback on localhost vs production origin; Chargily redirect URLs with and without trailing slashes; sequence lifecycle when user opens multiple tabs; wishlist FK cascade on user delete.

Test data hygiene. Use disposable Postgres schemas or docker-compose for CI, seed minimal categories/products, and rotate OAuth test users to avoid rate limits on Google’s consent screen during automated runs.

Chapter 19

Roadmap themes

  1. Hardening auth gates on admin paths
  2. Inventory reservation service
  3. Richer CMS for blogs
  4. Assistant evaluation dashboard
  5. Webhook-driven order fulfillment status

Each theme should ship with measurable KPIs: e.g., assistant NDCG@k on held-out queries, checkout conversion rate by wilaya, median time-to-first-byte on PDP, and error budget for payment callbacks.

Chapter 20

Internationalization, locales, and assistant copy decks

Vitrina’s customer-facing UI is predominantly French/Arabic/Darija-friendly in merchandising copy, while the assistant route codifies four response locales—Modern Standard Arabic, French, Algerian Darija (Latin), and English—selected by script detection, provider-reported language, and regex heuristics tuned for Maghrebi French and Darija markers.

Template banks centralize operational messages (missing API keys, rate limits, empty queries, catalog-only fallback). This pattern keeps UX consistent even when the LLM narrative varies, and simplifies legal review of deterministic strings.

Future i18n work could externalize UI strings into message catalogs (e.g., next-intl) while preserving assistant locale detection as a separate concern from static chrome translation.

Chapter 21

Repository layout and layering conventions

The repository follows a pragmatic vertical slice: src/app for routing and route handlers, src/components for UI modules grouped by domain (Shop, Checkout, Cart, Common), src/server for auth/db/data-access, src/redux for client state, and src/lib for cross-cutting utilities. This mirrors how teams onboard: start at a route, descend into components, drop to server helpers when persistence is involved.

Keeping Drizzle schema centralized avoids drift between “types” and “truth”. Migrations in drizzle/*.sql should be treated as immutable history once applied to production.

Chapter 22

Catalog data model and shop presentation

Products are relational rows with slugs for clean URLs, joined categories, and optional promotional pricing. The storefront also consumes curated shopData for deterministic assistant retrieval—an intentional dual path: relational data for transactional integrity, curated JSON for fast LLM grounding without N+1 queries at inference time.

Merchandising components emphasize photography, manufacturer, and category breadcrumbs to reduce uncertainty before add-to-cart.

Chapter 23

Product detail page: composition and assistants

The PDP composes gallery, price modes, description, reviews, and contextual assistant entry points. Product-scoped assistant payloads include title, availability hints, and category to steer retrieval toward compatible substitutes and accessories.

Separating global assistant chrome from PDP assistant reduces accidental context leakage and improves relevance metrics in telemetry.

Chapter 24

Cart, persistence, and client state lifecycle

Redux holds authoritative client cart state; persistence middleware writes through to storage after meaningful transitions. Hydration strategies must avoid double-counting during SSR handoff—Vitrina’s approach favors client reconciliation with stable keys to prevent flicker when session state and anonymous carts meet at login.

Chapter 25

Checkout UX: wilaya, commune, shipping presentation

Algerian wilaya and commune fields align with last-mile carrier expectations. The checkout surface presents Yalidine Express branding in the shipping method presentation layer, signaling fulfillment partner intent to shoppers even when rate tables are simplified in early versions.

Chapter 26

Payments: Chargily integration principles

Server-created checkout sessions ensure secrets never leave trusted compute. Amounts and success/failure URLs should be derived from trusted cart recomputation on the server, not from client-submitted totals alone. Idempotency keys for order creation pair naturally with payment intent identifiers.

Chapter 27

Wishlist domain model and UX

Wishlists are modeled as one row per user with a join table for products, enabling efficient uniqueness constraints and cascade deletes. UX surfaces highlight saved items and nudge conversion through return visits.

Chapter 28

Reviews, ratings, and moderation considerations

Reviews bind to users and carry integer ratings and free-text comments. Moderation policies (profanity filters, report button, admin quarantine) are organizational choices; technically the schema supports append-only consumer history suitable for audit.

Chapter 29

Site feedback instrument

Holistic site feedback complements product reviews by capturing satisfaction with navigation, speed, and support. Aggregating feedback alongside assistant telemetry can explain macro conversion changes.

Chapter 30

Blog subsystem and editorial workflow

Blog routes provide editorial grid/detail layouts with optional sidebars for long-form storytelling—useful for launches, lookbooks, and merchant education. A headless CMS integration would be a natural evolution without changing public URLs.

Chapter 31

Legal, trust, and policy surfaces

Terms, privacy, and refund policies establish baseline consumer trust. These pages should be versioned in git with change logs when material terms shift (returns window, data retention, payment dispute flows).

Chapter 32

Performance engineering checklist

  • Image formats and sizes for hero and PDP galleries
  • Defer non-critical assistant bundles until interaction
  • Memoize expensive selectors; avoid rerendering entire grids on cart tick
  • Prefer server components for static policy pages
  • Monitor API cold starts on Vercel for assistant route
Chapter 33

Accessibility baseline

Semantic landmarks, focus management for modals (quick view, assistant), sufficient color contrast on navy backgrounds, and keyboard operability for primary CTAs are baseline expectations. Automated checks (axe) plus manual screen reader passes on checkout are recommended before major releases.

Chapter 34

Engineering process: migrations, typing, and CI

Adopt a migration-first workflow: local drizzle-kit generate after schema edits, peer review SQL, apply in staging, snapshot restore drills before production. TypeScript strictness should ratchet over time; new modules should avoid any in auth and payment boundaries.
